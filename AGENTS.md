# Nixul AGENTS.md

This file provides instructions for agentic coding agents working in this repository.

## Build, Lint, and Test Commands

```bash
# Format Nix files (2-space style)
nix fmt

# Lint for unused variables
nix develop --command deadnix --fail .

# Run all checks/tests
nix flake check --all-systems --show-trace

# Build a specific host (dry-run)
nh os build .#<host>

# Test activation without applying
nh os test .#<host>

# Apply configuration (persist activation)
nh os switch .#<host>

# Rollback to previous generation
nh os rollback .#<host>
```

## Code Style Guidelines

### Language

- The codebase is primarily written in the **Nix language**
- Use 2-space indentation for Nix expressions

### Naming Conventions

- Files and directories: `kebab-case` (e.g., `my-module.nix`)
- Attribute sets: `camelCase` or `kebab-case` consistently within modules
- Option names: use kebab-case

### Imports and Modules

- Inputs are managed in `flake.nix`
- Use flake-parts for composition
- Auto-imports via `nix/lib/import-tree.nix` import all `.nix` files in subdirectories
- Each module defines its own options and gating

### Types and Pattern Matching

- Use pattern matching on attribute sets for function arguments to ensure required values are present
- Validate required fields before proceeding

### Error Handling

- Use `builtins.tryEval` for graceful failures
- Provide meaningful error messages with `throw`
- Fail fast with clear messages for configuration errors

### Comments

- Add comments to explain complex logic, not to describe what the code does
- Document non-obvious decisions and workarounds

### Commits

- Follow conventional commit standards: `type(scope): description`
- Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

## Module System

### Structure

The codebase uses a custom module system located in `nix/modules`.

### Directory Layout

- **`apps/`**: User-facing programs (browsers, media, gaming, AI)
- **`core/`**: Baseline OS, security, users, time, networking defaults
- **`desktop/`**: Display stack, compositors, panels, theming
- **`dev/`**: Developer tools, runtimes, editors
- **`hardware/`**: Drivers, audio, power, inputs, storage
- **`services/`**: Long-running daemons, databases, VPNs

### Rules

1. **Limit Items:** Keep folders at roughly 5 items max. Split into subfolders if it grows larger
2. **Reuse:** Prioritize reusing existing folders over creating new ones
3. **`default.nix`:** Only use for ordering or shared configuration; otherwise rely on auto-imports
4. **Metadata:** Write proper module metadata, not autogenerated
5. **Integration:** After adding a new module, add it to relevant hosts or users

### Module Contract

Module files can return:

- `options`: Generated option definitions
- `system`: NixOS system configuration
- `home`: Home Manager configuration

### Generated Option Roots

- `nixul.host.modules.<path>` for host-side modules
- `nixul.users.<name>.modules.<path>` for per-user modules

## Themes

- Load from `nix/nixul/themes/*` with `(loadTheme "<name>")` in host imports

## Architecture Flow

1. `flake.nix` calls `nix/lib/default.nix` to generate one `nixosSystem` per host directory
2. Base modules: `nix/nixul`, `nix/hosts/<host>`, Home Manager wiring, and `nix/lib/module-importer.nix`
3. Hosts set `nixul.host.modules`; users set `nixul.users.<name>.modules`
4. Importer discovers `nix/modules/**` and maps each file path to an option path
5. Importer evaluates each module's `system` and `home` functions with that module's `cfg` subtree

## Host and User Model

- Hosts are thin and focus on machine identity plus module selection
- Host fields are under `nixul.host`
- User definitions live under `nix/users/*` and populate `nixul.users.<name>`
- `nixul.primaryUser` is set from user entrypoints

## Universal Subsystems

- **Keybinds:** `nix/nixul/universal/keybinds/**` with Hyprland and Niri adapters
- **Bookmarks:** `nix/nixul/universal/bookmarks/**` for browser/dashy/glance outputs
- **Aliases:** `nix/nixul/universal/aliases/**`
- **Secrets:** `nix/modules/core/security/secrets/sops.nix` with encrypted data under `nix/assets/secrets/`

## Secrets Management

- Secrets are encrypted in-repo using `sops-nix`
- Encrypted files stored in `nix/assets/secrets/`
- Set `SOPS_AGE_KEY_FILE` for decryption
- Never commit decrypted material

## Operational Workflow

1. `nix fmt` - Format code
2. `deadnix` - Lint for unused variables
3. `nix flake check` - Validate configuration
4. `nh os test .#<host>` - Test without applying
5. `nh os switch .#<host>` - Apply configuration

## Editing Expectations

- Follow existing code conventions in the file being edited
- Prefer modifying `nix/modules/**` or `nix/users/**` before adding host-specific duplication
- Keep Nix formatting compatible with `nix fmt`
- Validate with lint and flake check after making changes

## Common Pitfalls to Avoid

- Do not use the old `import-tree.nix` auto-import architecture; use `module-importer.nix`
- Do not reference old flat option fields like `nixul.user` or `nixul.email`; use `nixul.host.*` and `nixul.users.*`
- Do not assume host-specific `home.nix` files are the primary pattern
- Do not claim all services are enabled by default; per-host overrides decide that
